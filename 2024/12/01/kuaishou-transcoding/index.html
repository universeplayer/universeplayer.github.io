在快手做ML：54万核集群教会我的事

发表于2024-12-01|更新于2026-03-02|Internship
|总字数:1.6k|阅读时长:4分钟|浏览量:

第一天：代码库比我整个大学写的都多

2024 年 7 月，我入职快手音视频架构部，做视频转码的分发策略算法。

第一天 mentor 跟我说：”先熟悉下代码。” 然后给我指了个 GitLab 仓库。

我点进去一看——好家伙，几万行代码，目录结构嵌了七八层，光 feature engineering 的文件就有几十个。我当时的表情大概和第一次打开 Linux 内核源码的人差不多：表面镇定，内心在尖叫。

在学校的时候我以为自己写过不少代码了（ACM 选手嘛，刷题刷了几千道），但那些都是几百行解决一个问题的东西。工业级代码库是另一个次元——你不光要看懂逻辑，还要搞清楚每个模块跟上下游系统怎么通信，特征从哪来的，数据流怎么走的，线上服务延迟要求是多少……

第一周我基本在读代码 + 问蠢问题中度过的 😅

54万核：数字大到失去实感

然后 mentor 给我讲了下业务背景，我就被震住了。

快手的视频转码集群有超过 50 万核，分发带宽峰值超过 50Tbps。

这是什么概念？说白了就是，你做的每一个决策——这条视频要不要转高清——背后都是成百上千台机器在跑。一个百分点的优化就是几百万人民币的成本变化。一个 bug 上线可能导致几千万用户看到模糊视频或者公司多花几百万带宽费。

我当时心想：我一个实习生，真的能碰这个吗？？？

但很快我就被扔到了具体的活上——优化 XGBoost 模型，决定”对哪些视频该转高清”。核心思路是预估每条视频转高清后的净收益（体验提升的商业价值减去转码和带宽成本），然后按阈值做决策。

调参地狱

说到 XGBoost 调参，我真的有段 PTSD。

特征工程倒还好——用户设备分辨率、网络环境（WiFi 还是 4G）、视频垂类（风景旅拍对清晰度敏感，对话类就还好）、作者历史热度、视频发布后前几分钟的播放增速……这些维度我理解得还算快。

但调参是真的折磨人。learning rate、max_depth、min_child_weight、subsample、colsample_bytree……每个参数都有一堆讲究，组合起来就是指数级的搜索空间。

我前前后后调了两百多组参数。Wandb 上的 run 从 exp_001 一路排到 exp_250，每跑一组都要等好一会儿，然后盯着 AUC 的变化。有时候 AUC 涨了 0.1%，你不知道是真的涨了还是噪音。有时候换了个参数 AUC 掉了但线上可能反而好——因为你优化的真正目标是净收益，不是 AUC。

那段时间我最常做的动作就是：提交实验 → 等结果 → 叹气 → 改参数 → 再提交。属于是薛定谔的调参，不看结果之前每组参数都充满希望 🎰

那次尴尬的 A/B Test

最让我印象深刻的教训来自一次 A/B test。

我改了特征工程，加了几个新特征，离线评估上 AUC 涨了 5%——这在离线指标里已经是很大的提升了。我兴奋地跟 mentor 说可以上线了。

上线跑了一周 A/B test，结果出来：完全没有显著差异。

啊？？？

排查了好久才发现原因：我新加的一个特征在离线训练数据里是完整的，但在线上的实时特征服务里有几分钟的延迟。也就是说模型训练的时候用的是”未来信息”，但线上推理的时候用的是过时数据。经典的 offline-online gap。

Mentor 拍了拍我的肩说：”这种坑，每个做推荐系统的人都要踩一遍。”

说实话那个时候挺受打击的。但后来想想，这种教训在课本上是怎么都学不到的。从那以后我每次加新特征第一件事就是确认线上能不能实时拿到，延迟是多少。听起来很基础对吧？但真的，只有踩过坑的人才会养成这个习惯。

删 1000 PB 数据：小心翼翼如履薄冰

实习后期我还参与了存储治理项目——清理历史冗余的转码文件。

快手每天产生海量的视频转码产物，同一个视频可能有好几个分辨率、好几种编码格式的副本，很多早就没人看了。我们的任务是把这些冗余数据安全地清掉。

最终清理了超过 1000PB。

这个数字听着很爽，但过程是真的提心吊胆。我们设计了三步走的策略：先标记超过 N 天没人看的副本，然后从热存储迁到冷存储观察一周，确认没有回源请求才真正删除——而且只删转码副本，原始视频永远不动（最坏情况可以重新转码）。

每次执行删除脚本之前我都要检查三遍 SQL 条件，生怕把不该删的删了。Mentor 说”删错一批数据你就知道什么叫写事故报告了”。这句话现在想起来都后背发凉 😨

快手教会我的

半年时间不长，但这段实习彻底改变了我对”做 ML”的理解。

在学校里做 ML 是这样的：找数据集 → 跑模型 → 看指标 → 写论文。clean and simple。

在快手做 ML 是这样的：理解业务 → 搞清楚特征从哪来 → 确认线上服务能不能用 → 调参 → 离线评估 → 上 A/B test → 分析结果 → 排除噪音 → 可能发现离线涨了线上没涨 → 排查原因 → 重新来 → 终于显著了 → 灰度发布 → 全量上线 → 继续监控……

模型本身可能只占整个工作量的 20%，剩下 80% 是工程、是数据、是对业务的理解。

以前觉得”ML Engineering”就是个 title，现在才知道这是一个完整的、复杂的、和纯粹的”ML Research”截然不同的技能体系。

在快手我终于理解了什么叫工业级 ML。 不是你的模型多花哨，而是你的整个系统在 54 万核的集群上、56 Tbps 的带宽里、3 亿日活用户面前，能稳定、高效、可靠地运行。这才是真功夫。
