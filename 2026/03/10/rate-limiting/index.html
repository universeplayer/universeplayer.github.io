高并发系统的限流：四种算法的选择逻辑

发表于2026-03-10|更新于2026-03-05|技术
|总字数:1.5k|阅读时长:4分钟|浏览量:

限流是每个做后端的人绕不开的话题。它不难理解，但选哪种算法、在哪个层面做、怎么在分布式环境下实现原子性，这些细节搞清楚之前，坑踩起来很疼。

为什么需要限流

先说动机。系统有处理能力上限，超过这个上限，要么响应时间飙升，要么直接崩溃。更危险的是雪崩效应：上游流量洪峰打垮 A 服务，A 服务超时导致 B 服务积压，B 服务崩溃触发 C 服务重试……整条链路全垮。

限流的本质是在系统入口做流量整形，把超出处理能力的请求拒绝掉或排队等待，而不是让它们全部涌进来把系统压死。代价是部分请求会被拒绝，但这比全部请求都超时或者系统直接崩掉要好得多。

四种算法

固定窗口

最简单的实现：把时间划分成固定大小的窗口（比如每秒一个窗口），每个窗口内维护一个计数器，超过阈值就拒绝。

实现简单，Redis 里一个 INCR 加 EXPIRE 就能搞定。但有个著名的边界问题：如果阈值是 100 req/s，在第 0.9 秒来了 100 个请求（没超限），在第 1.1 秒又来了 100 个请求（新窗口，也没超限），但从 0.9s 到 1.1s 这 0.2 秒内实际上有 200 个请求通过了。边界突刺可以达到阈值的两倍，对下游来说是个冲击。

滑动窗口

解决边界突刺问题的方案。不再是固定的时间格，而是以当前时刻为终点，往前看固定长度的窗口，统计这个窗口内的请求数。

实现上可以用 Redis 的 Sorted Set，以时间戳为 score 存储请求记录，查询时用 ZRANGEBYSCORE 统计时间窗口内的请求数，过期的记录用 ZREMRANGEBYSCORE 清除。准确，但比固定窗口消耗更多内存，每次请求都要做 set 操作而不是单纯的 INCR。

漏桶

把请求想象成往桶里倒水，桶底有一个固定速率的出口。无论入口的流量多么不均匀，出口总是以恒定速率放行请求。桶满了就溢出（拒绝请求）。

优点是输出极度平滑，对下游非常友好。缺点是即使系统当前完全空闲，也不能利用突发流量——桶里的水只能按固定速率流出。对于允许短时高并发的场景（比如秒杀活动的预热期），漏桶不是好选择。

令牌桶

令牌以固定速率放入桶中，每个请求消耗一个令牌，桶满时令牌不再累积。请求到来时，有令牌就处理，没有令牌就拒绝（或等待）。

关键区别在于令牌可以积累。系统空闲时，桶里攒满令牌；流量突发时，可以瞬间消耗积累的令牌，允许短时超过平均速率的流量通过，只要总量不超过桶容量。这对于有正常流量基线但偶尔有合理突发的场景非常合适。

用咲夜举个例子。咲夜可以停止时间，在主观上静止的那段时间里，她在悄悄地往令牌桶里放令牌——摆好下午茶的茶杯、准备好瓷器、把茶壶摆在恰当的位置。时间恢复之后，可以瞬间接待远超平时数量的客人，因为准备工作已经提前做完了。令牌桶就是这个逻辑：平静期积累处理能力，突发期集中释放。漏桶没有这个能力，不管之前多空闲，每秒只能接待固定数量的客人。

分布式环境下的额外麻烦

单机限流简单，分布式集群就复杂了。每个实例独立维护状态，全局限流就需要中心化存储，Redis 是最常用的方案。

限流可以做在不同的层面，逻辑不同：

Nginx/Gateway 层限流的优势是在请求进入应用之前就拦截，节省应用服务器资源。Nginx 的 limit_req_zone 模块直接支持漏桶算法，配置简单。缺点是粒度粗，只能基于 IP 或 URL 粒度，无法感知业务语义（比如按用户等级设置不同的限流阈值）。

应用层限流可以做到更细粒度：特定用户、特定接口、特定资源类型分别设置不同的策略。代价是每个请求都要有额外的 Redis 查询开销，以及代码复杂度增加。

实际项目里两层都做是合理的：Nginx/Gateway 处理明显的流量攻击和全局保护，应用层做业务级别的精细控制。不要想着只做一层就万事大吉。

选哪个

没有万能答案，看场景：

  • 实现简单、对突刺不敏感：固定窗口
  • 需要精确控制、可以接受内存开销：滑动窗口
  • 下游对流量平滑有严格要求：漏桶
  • 允许合理突发、最接近真实业务需求：令牌桶（大多数场景的推荐选择）

分布式场景下记得用 Lua 保证原子性，别让竞争条件悄悄绕过你的限流逻辑。
