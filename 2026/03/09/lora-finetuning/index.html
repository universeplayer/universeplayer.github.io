LoRA：用0.1%的参数微调LLM，不是玄学是线代

发表于2026-03-09|更新于2026-03-05|技术
|总字数:1.8k|阅读时长:5分钟|浏览量:

先说一个类比

十六夜咲夜管理着整个红魔馆的时间流和日程。如果蕾米莉亚今天突然有新的要求——比如”下午茶改在花园而不是大厅”——咲夜不需要重新组织整个红魔馆的运转系统，她只需要在原有的日程表上做一个小小的调整：把”大厅”这一项改成”花园”，其他一切照常。

LoRA 干的就是这件事：不改动预训练模型的全部参数，而是给每一层的权重矩阵附加一个”小小的调整量”，用这个调整量来承载新任务的信息。

这听起来很简单，但为什么它能 work？这背后有一个具体的数学依据。

全参数微调的问题

一个典型的 LLM（比如 LLaMA-7B）有 70 亿个参数。对它做全参数微调（Full Fine-tuning）意味着在训练过程中所有 70 亿个参数都在更新，需要存储梯度、优化器状态（Adam 的话需要一阶矩和二阶矩，参数量 ×3），显存消耗大致是模型本身的 4 倍以上。一个 7B 模型的全参数微调，在 float32 下需要超过 100GB 显存，这对绝大多数人来说是不可能的。

LoRA 的参数量是什么量级？如果对所有 Transformer 层的 Q、K、V、O 矩阵都加 LoRA，rank 取 8，7B 模型的可训练参数大约是 4-40M，也就是总参数量的 0.05%-0.5%。显存降低到可以在消费级 GPU 上跑。

但参数少了这么多，效果怎么样？很多任务上 LoRA 微调的效果和全参数微调接近，有些任务上甚至更好（因为参数约束起到了一定的正则化效果，防止过拟合）。这就是它值得深入理解的地方。

低秩分解：数学上到底在做什么

LoRA 的核心假设是：模型在适配新任务时，权重矩阵的变化量 $\Delta W$ 是低秩的（low-rank）。

形式化地写，对于原始权重矩阵 $W_0 \in \mathbb{R}^{d \times k}$，LoRA 的参数化方式是：

$$W = W_0 + \Delta W = W_0 + BA$$

其中 $B \in \mathbb{R}^{d \times r}$，$A \in \mathbb{R}^{r \times k}$，$r \ll \min(d, k)$。$r$ 就是所谓的 rank。

原始 $\Delta W$ 有 $d \times k$ 个参数。拆成 $B$ 和 $A$ 之后，参数量变成 $r(d+k)$。当 $r=8$，$d=k=4096$ 时，参数量从 16M 降到约 65K，降了两个数量级。

训练时 $W_0$ 冻结（frozen），只训练 $A$ 和 $B$。初始化时 $A$ 用高斯随机初始化，$B$ 全零初始化，这样训练开始时 $\Delta W = BA = 0$，保证微调起点和原始模型相同。

为什么更新矩阵是低秩的：内在维度假说

这是 LoRA 能 work 的理论基础。

Aghajanyan et al., 2021（《Intrinsic Dimensionality Explains the Effectiveness of Language Model Fine-Tuning》）在 LoRA 之前做了一个实验：他们用随机投影把梯度更新约束到一个低维子空间里，然后看在不同维度下微调能达到什么效果。结论是：对于大多数 NLP 任务，模型微调所需的”内在维度”（intrinsic dimension）远远小于参数总量——很多任务的内在维度在几十到几百的量级，而参数量是几十亿。

直觉解释：预训练模型已经在大规模数据上学到了丰富的表示，适配一个具体的下游任务，本质上是在已有表示空间里做一个小的”转向”，而不是从零开始学一个新的方向。这个转向所需的自由度很少，因此权重更新的有效信息可以被一个低秩矩阵近似捕获。

换句话说：模型在预训练后，参数空间里大部分方向已经是”有意义的”，微调需要移动的只是其中一个低维的流形。

Rank 的选择

rank $r$ 是 LoRA 的核心超参数，选多大是有讲究的。

原始论文（Hu et al., 2022）里的发现是：在很多任务上 $r=4$ 或 $r=8$ 就已经足够好，继续增大 rank 边际收益递减，而且过大的 rank 会失去低秩约束的正则化效果，在小数据集上反而更容易过拟合。

经验规则：

  • 简单的指令跟随任务（格式、风格调整）：$r=4$ 或 $r=8$
  • 领域适应（让模型了解特定领域的知识和术语）：$r=16$ 或 $r=32$
  • 复杂任务（代码、数学推理这类需要学习新技能的任务）：$r=64$ 或更高

但这些都是经验性的，最终还是要在你的数据集上做消融实验。

QLoRA：量化 + LoRA

QLoRA（Dettmers et al., 2023）在 LoRA 的基础上加了一步：把冻结的基础模型参数用 4-bit 量化（NF4 量化，专门为正态分布设计的量化格式）存储，只有可训练的 LoRA 参数保持 float16 精度。

这把内存消耗进一步压缩，让 65B 参数的模型可以在单张 48GB GPU 上微调。代价是计算时需要把 4-bit 权重反量化回 float16 做前向传播，有一定的额外计算开销，但这个开销通常是可接受的。

工程上的坑

合并权重时的精度损失。训练完成后，LoRA 权重可以合并回基础模型：$W \leftarrow W_0 + BA$。这个加法操作如果在 float16 下进行，对于基础权重数值很大而 $BA$ 很小的情况，会有一定的精度损失。推荐在 float32 下做合并，然后再转回 float16 存储。

不同任务的 rank 需求差异很大。一个为代码生成微调的 LoRA 和一个为情感分类微调的 LoRA，最优 rank 可能相差 10 倍。跨任务复用同一套 LoRA 超参数不一定合适，需要针对任务做调整。

LoRA 加在哪些层。原始论文只对 Attention 的 Q 和 V 矩阵加 LoRA。后续研究发现对 Q、K、V、O 以及 FFN 的 up_proj 和 down_proj 都加 LoRA，效果更好，但参数量也更大。没有通用答案，看任务和资源预算。

学习率比全参数微调高。LoRA 的参数量小，如果用全参数微调的学习率（比如 1e-5），更新量太小收敛很慢。通常 LoRA 的学习率要调到 1e-4 甚至更高。

一句话总结

LoRA 不是参数压缩的黑魔法，它的有效性有具体的理论支撑：预训练模型的权重更新有内在低秩结构，只要用低秩矩阵近似这个更新，参数量可以大幅降低而不损失太多表达能力。

具体参数怎么选，还是要做实验。没有万能的 $r=8$，但 $r=8$ 是一个合理的起点。
