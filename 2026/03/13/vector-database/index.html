向量数据库原理：HNSW和IVF-PQ

发表于2026-03-13|更新于2026-03-05|技术
|总字数:2.1k|阅读时长:7分钟|浏览量:

为什么KNN不够用

最近邻搜索的朴素算法是精确KNN：给定查询向量 $q$，在数据库里遍历所有向量 $x_i$，计算距离 $d(q, x_i)$，返回最小的K个。

时间复杂度 $O(nd)$，其中 $n$ 是向量数量，$d$ 是向量维度。

听起来无害，算一下：$n = 10^8$（亿级文档），$d = 1536$（OpenAI embedding的维度）。单次查询需要 $1.536 \times 10^{11}$ 次浮点运算，A100每秒大约 $3 \times 10^{14}$ 次浮点运算（理论峰值），实际上一次查询要几十到几百毫秒。

多一点并发，延迟就不可接受了。而且这是最好情况，实际的KNN实现里还有内存读取的瓶颈——亿级向量的数据集放在内存里本身就是十几到几百GB，每次查询都要线性扫描这么大的数据，IO是真正的瓶颈。

所以生产系统里用的是ANN（Approximate Nearest Neighbor，近似最近邻）：允许一定的精度损失（召回率99%而非100%），换取查询速度的显著提升。两个主流算法是HNSW和IVF-PQ，它们用完全不同的思路解决同一个问题。

HNSW：分层跳表加小世界图

HNSW（Hierarchical Navigable Small World，Malkov & Yashunin, 2018）的设计思想来自两个地方：跳表（Skip List）和小世界图（Small World Graph）。

小世界图

小世界图是一种特殊的图结构：任意两个节点之间的平均路径长度是 $O(\log n)$，同时有较高的局部聚类系数。现实世界里的很多网络（社交网络、引文网络）都有这个性质——你和任何陌生人之间平均只有六度分隔，但你的朋友圈里互相认识的比例很高。

在向量空间里，如果把相近的向量连边构成这样的图，那从任意一个起点出发，通过贪心地每次走向离查询最近的邻居，能在 $O(\log n)$ 步之内到达目标区域。这是NSW（Navigable Small World）的基本思路。

加上层次结构

NSW的问题是：对于大数据集，查询路径虽然短，但每一步的候选邻居很多，搜索效率不够高。

HNSW加了一个层次化的结构，类比跳表：

  • 最底层（layer 0）包含所有向量，每个向量和最近的若干邻居相连
  • 上层是底层的”快速通道”：向量以指数递减的概率出现在更高层，每个向量在第 $l$ 层出现的概率是 $e^{-l/m_L}$（$m_L$ 是归一化因子）
  • 每层内部维护局部小世界图结构

查询时，从最顶层的单个入口点出发，贪心地在当前层找到离查询最近的节点，然后进入下一层继续细化，直到到达底层。

查询复杂度：$O(\log n)$（层数 $O(\log n)$，每层内搜索步数 $O(1)$ 均摊）。

插入一个新向量的复杂度也是 $O(\log n)$：随机决定它出现在哪几层，在每层找到它的邻居并更新边。这使得HNSW支持动态增量更新，不需要重建索引。

HNSW的代价

HNSW的内存消耗是它最大的问题。每个向量存原始向量（$d \times 4$ bytes，float32）加上图结构（每层的邻居列表，$O(m)$ 条边，$m$ 是每个节点的最大邻居数）。

对于 $n = 10^8$，$d = 128$，$m = 32$ 的配置，内存占用大约 $100GB$ 以上。这在很多场景是不可接受的。

IVF-PQ：聚类分桶加向量压缩

IVF-PQ用两个独立的技巧解决HNSW的两个问题：IVF减少搜索范围，PQ压缩内存。

IVF：Inverted File Index

IVF的思路类似倒排索引，但作用于向量空间：

  1. 离线训练阶段：用K-Means对数据库向量聚类，得到 $K$ 个聚类中心（称为码本，codebook）
  2. 建索引：每个向量找到最近的聚类中心，分配到对应的桶（list）里
  3. 查询时：先找到查询向量最近的若干个桶（通常是 nprobe 个），只在这些桶里做精确搜索

搜索范围从全部 $n$ 个向量缩小到 $K$ 个桶里平均每桶 $n/K$ 个向量，加上 nprobe 个桶：实际搜索量约 $\text{nprobe} \times n / K$。

代价：nprobe 越大，召回率越高但速度越慢；聚类质量影响召回率；数据分布不均匀时某些桶特别大，影响性能。

PQ：Product Quantization

PQ（乔丁 & 比耶罗, 2011）解决内存问题：用量化压缩每个向量。

把 $d$ 维向量切分成 $m$ 个子向量，每段 $d/m$ 维。对每段分别训练一个码本，有 $K^$ 个中心（通常 $K^ = 256$，用1字节表示）。

每个向量压缩存储为 $m$ 个字节（每段的量化码），从原来的 $d \times 4$ bytes 压缩到 $m$ bytes。

压缩比：$d = 128$，$m = 8$ 的情况下，从512 bytes压缩到8 bytes，64倍压缩。

查询时，距离不直接在原始向量上算，而是用预计算的距离查找表（ADC，Asymmetric Distance Computation）来近似：

$$d(q, x) \approx \sum_{j=1}^{m} d\left(q_j, c_j^{(q_j)}\right)$$

其中 $c_j^{(i)}$ 是第 $j$ 段码本的第 $i$ 个中心，$q_j$ 是查询向量第 $j$ 段的量化码。这个距离是近似的，因为向量被量化了，会有误差。

两者对比

特性 HNSW IVF-PQ
查询速度 快 ($O(\log n)$) 快（取决于nprobe）
召回率 高 中（有量化误差）
内存占用 大（原始向量+图结构） 小（压缩存储）
动态更新 原生支持 需要重建量化器
训练成本 无（直接插入） 需要离线训练聚类和量化
适用场景 中小规模，高精度 大规模，内存受限

选型的粗略规则：向量数量在1亿以下、内存足够、需要动态更新，选HNSW；向量数量超过1亿或者内存受限、更新不频繁，选IVF-PQ。

工程选型：各数据库的定位

Faiss（Meta开源）：研究和工程都用，实现了HNSW、IVF-PQ、IVFFLAT等主流算法，C++实现、Python绑定，GPU版本也有。不是一个完整的数据库，是个算法库，适合需要精细控制的场景或者研究用途。

Qdrant（Rust实现）：完整的向量数据库服务，默认用HNSW，支持过滤查询（按条件过滤后再做ANN），API设计比较现代，适合中小规模生产部署。

Weaviate：有类似图数据库的查询语义，支持混合搜索（向量+关键词），适合需要把结构化属性和向量查询结合起来的场景。

Milvus：云原生向量数据库，专为大规模设计，支持分布式部署，Alibaba和Zilliz在背后。大规模生产场景（十亿级向量）是它的主场。

RAG场景的特殊需求

RAG系统对向量数据库有一些普通ANN场景没有的要求：

动态增量更新。文档库会持续更新，新的文档要能实时加入索引，不能等到离线批量重建。这让HNSW比IVF-PQ更自然——IVF-PQ的量化器是离线训练的，新加的向量要重新量化才能索引，如果量化器不覆盖新向量的分布，精度会下降。

过滤查询。RAG经常需要”在满足条件X的文档里找最相关的”，比如只在某个用户的私有文档里搜。纯ANN索引不直接支持这个，需要ANN加上后处理过滤（先ANN再过滤，可能损失召回率），或者支持pre-filtering的数据库（Qdrant的payload indexing就是做这个的）。

美琴的做法是：面对一堆嫌疑人，不是挨个审问，而是先按电系能力、发型、制服划出几个群体（IVF），然后在最可疑的那几群里精确比对（PQ或者HNSW底层搜索）。找不到嫌疑人不要紧，95%的情况下那个人在她锁定的范围内，剩下5%是小概率事件，可以接受。

这就是ANN的精髓：用可接受的精度换不可接受变成可接受的速度。

参考文献：Malkov & Yashunin, “Efficient and robust approximate nearest neighbor search using HNSW graphs”, IEEE TPAMI 2018. Jégou et al., “Product Quantization for Nearest Neighbor Search”, IEEE TPAMI 2011.
