二分查找：最简单也是最容易写错的算法

发表于2026-03-06|更新于2026-03-04|算法
|总字数:1.8k|阅读时长:6分钟|浏览量:

Knuth 在《The Art of Computer Programming》里提到，二分查找的思想 1946 年就有了，但第一个完全正确的实现直到 1962 年才出现。中间隔了 16 年。[Knuth, TAOCP Vol.3, Section 6.2.1]

这不是一个关于”聪明人花时间才解决的难题”的故事，而是关于”细节有多磨人”的故事。二分查找的思想任何人五分钟能懂，边界条件能让任何人写出 bug。

为什么容易错

问题描述很简单：有序数组 a[0..n-1]，找目标值 target 的位置，O(log n)。

但”找”这个字其实有三种意思：

  1. 找 target 是否存在（精确查找）
  2. 找第一个 >= target 的位置（下界，lower bound）
  3. 找最后一个 <= target 的位置（上界，upper bound）

这三种写法的边界条件完全不同，不能混用。大部分写错的二分都是把这三种情况的边界条件拼在了一起。

另外还有一个经典 bug：

当 lo 和 hi 都接近 INT_MAX 时，lo + hi 溢出。C++ 里这是未定义行为，Java 里是直接得到负数。这个 bug 在 JDK 的 Arrays.sort() 里存在了将近十年，2006 年才修。

统一框架：用不变量思考

与其记三套写法，不如用一个框架推导所有情况：

核心思想：维护一个循环不变量。每次迭代结束后，答案一定在 [lo, hi] 里（闭区间或半开区间，取决于写法）。当 lo == hi 时，答案确定，终止。

以查找”第一个 >= target 的位置”为例推导：

  • 区间含义：a[0..lo-1] 都 < target，a[hi..n-1] 都 >= target
  • 初始状态：lo = 0, hi = n（注意 hi = n，不是 n-1，因为答案可能是 n，表示不存在）
  • 每次迭代：取中点 mid，如果 a[mid] < target，收缩左边界 lo = mid + 1；否则收缩右边界 hi = mid
  • 终止：lo == hi，答案就是 lo

关键细节：为什么 hi = mid 而不是 hi = mid - 1？因为我们不确定 a[mid] 是不是第一个，只能排除”mid 右边不是第一个”，mid 本身还得留着。

三种写法完整代码

注意第二种写法里 mid = lo + (hi - lo + 1) / 2 多了个 +1，这是向上取整。如果不加，当 lo = 0, hi = 1 时 mid = 0，执行 lo = mid 后 lo 没变，死循环。

这就是为什么说二分容易错——取整方向和收缩方向必须配套，否则就会死循环。

二分答案：更考验思维的用法

上面是在数组里二分，但二分更有价值的应用是”二分答案”：当答案本身具有单调性时，直接在答案空间里二分。

典型题型：最小化最大值，最大化最小值。比如”把数组分成 k 段，最小化各段最大和”——答案 x 越大，越容易满足（可以分成 k 段）；答案越小，越难满足。单调性成立，二分答案。

这种思路是 ACM 里相当常见的，很多看起来很复杂的问题一旦找到单调性就能用二分答案秒。

御坂美琴找线索

想象美琴在找一个她确定存在但不知道位置的线索，线索在一条有 1000 个房间的走廊里。她每次可以打开一个房间看看，怎么最快找到？

暴力法：从头到尾一个个找，最坏要开 1000 次。

二分：先开 500 号房间，根据某个判断条件决定线索在左边还是右边，然后去找 250 或 750，继续……最多 10 次（$\lceil \log_2 1000 \rceil = 10$）就能确定。

美琴每次排除掉一半的可能性，10 次就把 1000 个候选缩到 1 个。这就是 O(log n) 的直观含义——每次操作将问题规模减半，总操作次数是 $\log_2 n$。

当然，美琴真正的优势是可以直接用电磁波扫描走廊，不需要二分，但这不在今天的讨论范围里。

做题 Checklist

拿到一道题，用二分之前问自己：

  1. 单调性在哪里？（必须存在单调性才能二分）
  2. 找精确值，还是找边界（第一个满足/最后一个满足）？
  3. 答案空间是什么？在数组下标上二分，还是在答案值上二分？
  4. 区间开闭：[lo, hi] 还是 [lo, hi)？用哪种就从头到尾保持一致
  5. mid 的取整方向和收缩方向有没有配套？

特别注意：如果你写 hi = mid - 1 的那种写法，初始 hi = n - 1；如果你写 hi = mid 的写法，初始 hi = n。混了就是 bug。

边界条件这种东西，不是聪明就能避免出错，是习惯了固定模式才能不出错。选一套写法练到肌肉记忆，比每次都重新推更可靠。

二分查找 1946 年有思想，1962 年才有正确实现。这 16 年里肯定有无数聪明人写了”差不多正确”的版本。差不多正确和完全正确之间，隔着的就是这些边界条件。
