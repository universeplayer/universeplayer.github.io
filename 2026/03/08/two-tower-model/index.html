推荐系统双塔模型：工业界最常用的召回架构

发表于2026-03-08|更新于2026-03-05|技术
|总字数:1.9k|阅读时长:6分钟|浏览量:

先从芙宁娜的审判庭说起

枫丹的审判庭有一个不那么明显的工程问题：每次开庭，法官需要快速从大量候选证人里匹配出和案件最相关的那几位。候选证人可能有几百个，全部问一遍不可能，你需要一个机制先快速筛出来最相关的一批，再精细审查。

推荐系统的召回层干的就是这件事。

全量商品可能有几千万，你不可能对每个用户-商品对都跑一个精排模型（那代价是灾难级别的）。召回层的任务是从几千万里快速筛出几千个候选，再交给精排和重排去细化。双塔模型（Two-Tower Model）是这个环节目前工业界用得最广的架构。

双塔结构：把用户和物品分开建模

双塔的核心思路非常直接：用两个独立的神经网络（”塔”）分别给用户和物品生成 embedding 向量，然后用向量相似度来衡量匹配程度。

形式化地写：

$$\hat{y}(u, i) = \cos(f_\theta(u),\ g_\phi(i)) = \frac{f_\theta(u) \cdot g_\phi(i)}{|f_\theta(u)||\ g_\phi(i)|}$$

其中 $f_\theta$ 是用户塔，$g_\phi$ 是物品塔，两者独立参数化，不共享权重。

用户侧特征通常包括：用户历史行为序列、人口统计属性（年龄、地区）、实时上下文（时间、设备）等。物品侧特征包括：物品 ID、类别、标签、统计特征（点击率、完播率）等。每一侧各自经过若干层 MLP 或 Transformer，最终输出一个固定维度的 embedding（通常 64 或 128 维）。

这个设计的工程价值在后面说，先理解为什么要把两侧分开而不合并成一个大网络。

为什么不能直接用一个大网络

直接把用户特征和物品特征拼在一起输入一个网络（Cross-network），理论上表达能力更强，但在工业级召回场景里根本跑不动。

原因是系统延迟约束。一次召回请求进来，要在几十毫秒内从几千万物品里选出几千个候选。如果用 Cross-network，每对用户-物品都要做一次前向传播，几千万次前向传播串行跑根本不可能在 deadline 之内完成。

双塔的关键设计是离线/在线分离：

  • 物品塔可以离线跑，提前把所有物品的 embedding 算好，存到向量数据库（Faiss、Milvus 等）。
  • 线上来了请求，只跑一次用户塔，得到用户 embedding。
  • 然后在向量数据库里做 ANN（Approximate Nearest Neighbor）检索，找最近的 Top-K 物品 embedding。

整个线上过程只有用户塔推理 + ANN 检索，延迟可以控制在几十毫秒以内。这就是双塔的核心工程优势：把计算量大的物品侧推到离线，线上只需要轻量的检索。

负样本采样：最容易踩坑的地方

双塔用对比学习训练。正样本是用户实际点击/购买的物品，负样本是用户没有互动的物品。训练目标是让正样本对的相似度高、负样本对的相似度低。

听起来简单，但负样本的采样策略极其重要，错了模型上线效果会很差。

最常见的方案是 In-batch Negative Sampling（批内负采样）：在一个 batch 里，对于用户 $u_i$，把同 batch 内其他用户的正样本物品 ${i_j : j \neq i}$ 当作负样本。这样不需要额外采样，计算开销很小。

损失函数通常用 Softmax Loss：

$$\mathcal{L} = -\frac{1}{|B|} \sum_{i \in B} \log \frac{\exp(s(u_i, i_i) / \tau)}{\sum_{j \in B} \exp(s(u_i, i_j) / \tau)}$$

其中 $s(u, i) = \cos(f(u), g(i))$，$\tau$ 是温度系数，$B$ 是 batch。

但 In-batch Negative Sampling 有一个系统性的偏差问题：流行物品在 batch 里出现的概率正比于它的曝光频率，所以它作为负样本被学到”更难”的次数更多，最终导致流行物品的 embedding 被系统性地压低，召回时流行物品的分数会偏低——这和真实情况相反（流行物品本来用户就更可能点）。

Google 在 Sampling-Bias-Corrected Neural Modeling（Yi et al., 2019）里给出了修正方案：对每个负样本按物品的曝光概率 $p_j$ 做修正：

$$s_{\text{corrected}}(u_i, i_j) = s(u_i, i_j) - \log p_j$$

减去 $\log p_j$ 相当于对高频物品的分数做降权惩罚，把采样偏差从分数层面修正掉。这个修正在实际系统里效果很明显，是工程实践里的标准操作。

另一个坑是曝光偏差（Exposure Bias）：没有曝光的物品不一定是用户不喜欢的，也可能只是系统从来没把它推出去过。如果把所有未曝光物品都当负样本，等于在训练一个”用户对没见过的东西不感兴趣”的模型，这在冷启动场景下会严重伤害新物品。部分系统会加一个”硬负样本”采样策略：从曝光但未点击的物品里采样负样本，而非从全量物品里随机采。

工程上的其他注意点

ANN检索的精度/召回权衡。工业上常用 Faiss 的 IVFPQ（倒排+乘积量化），量化会引入一定精度损失，ANN 的 Recall@K 通常在 95% 以上就算可以了，追求 100% recall 在工程上不合算（需要用精确检索，代价高得多）。

Embedding 更新频率。物品侧 embedding 是离线算的，但物品的热度和特征是在变的。如果物品 embedding 几天才更新一次，系统对近期热点物品的召回会滞后。实际做法是对”高热度物品”做增量更新（每小时一次），对长尾物品做日级别更新，分频率处理。

冷启动的天坑。新物品没有历史交互数据，物品塔的行为序列特征是空的，embedding 质量很差。常见解法：用物品的内容特征（标题、图片）初始化 embedding，或者用聚类把新物品映射到最近的已有物品 embedding 附近作为初始值。这些都是 heuristic，都不完美，冷启动是推荐系统永远的遗留问题。

双塔之后：架构的演化方向

双塔的局限在于用户和物品的交互只在最后的点积层发生，中间没有特征交叉。这在精度上是有损失的。

一个改进方向是 DCN（Deep & Cross Network）：在生成 embedding 之前显式地做特征交叉，提升表达能力。另一个方向是在 Rerank 阶段用更重的 Cross Attention 模型弥补召回层的交互不足。

但在实际系统里，召回层对响应时间的要求决定了它不能太重，双塔的工程效率优势在短期内很难被替代。权衡不可能三角（精度、效率、可扩展性）是推荐系统工程永远的主旋律。

召回层的价值常常被低估，大家的注意力都在精排。但召回率（相关物品有多少比例被召回了）决定了精排的天花板——你召不上来的物品，精排再强也看不到。双塔是这个环节目前工程上最可靠的选择，不是因为它最精妙，而是因为它简单、稳定、好 debug。
