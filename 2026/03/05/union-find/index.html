并查集：最优雅的数据结构没有之一

发表于2026-03-05|更新于2026-03-04|算法
|总字数:2k|阅读时长:6分钟|浏览量:

一个芙宁娜不想面对的问题

枫丹法庭来了一批证人，芙宁娜需要确认：证人A和证人B，是不是认识同一个人？

更具体地说——她手里有一堆”某某和某某认识”的记录，然后她要在庭审中实时回答查询：”这两个人有没有共同认识的朋友？”

Naive的做法是什么？对每次查询，从A出发做BFS/DFS，遍历所有A认识的人、A的朋友认识的人……然后看有没有碰到B的邻居。这个在稀疏图上可以，但证人很多、关系记录一直在新增的情况下，每次查询都要O(V+E)，法庭效率会低到芙宁娜当场摔锤子。

这就是动态连通性问题（Dynamic Connectivity Problem）的基本形态：支持两种操作——

  • union(u, v)：将u和v所在的连通分量合并
  • find(u)：查询u所在连通分量的代表元素（通常用来判断u和v是否连通）

我们需要一个数据结构，让这两个操作都尽可能快。

基本想法：用森林维护连通关系

并查集（Union-Find / Disjoint Set Union，简称DSU）的核心思想很直接：

每个连通分量用一棵树来表示，树的根是这个分量的”代表”。每个节点只需要记住自己的父节点是谁。判断u和v是否连通，只需要看他们是否有同一个根。

这就是这个数据结构的全部了。其余的都是优化。

最基础的实现：

初始状态，每个节点的父节点是自己，即每个人都是独立的连通分量。

朴素实现的问题

朴素实现有一个严重的退化情形：如果每次都把x的根挂到y的根下面，树可能退化成一条链。

比如依次执行 unite(0,1), unite(1,2), unite(2,3), …，树就变成了：

这时候 find(0) 要遍历整条链，时间复杂度退化成O(n)。n次操作就是O(n²)，不可接受。

两个优化可以解决这个问题，而且可以叠加使用。

优化一：路径压缩（Path Compression）

路径压缩的直觉非常好理解：

公司里有一个很深的组织架构，你是基层员工，你的老板的老板的老板的老板才是CEO。现在CEO说：”以后你们都直接向我汇报，把中间层去掉。”这样下次查询谁是CEO，直接一跳就到了。

在 find 操作里加一行：

parent[x] = find(parent[x]) 这个赋值，在递归找到根的同时，把沿途所有节点的父节点直接更新为根。一次 find 之后，整条路径上的节点全部直接挂到了根上。

这一行是真正的精华，也是为什么我说并查集优雅——一行赋值，把路径压缩和查找操作合并了。

优化二：按秩合并（Union by Rank）

路径压缩解决了”已经建好的树太深”的问题，但没解决”合并时新树变深”的问题。

按秩合并的思路是：合并两棵树时，把矮的接到高的下面，而不是反过来。这样树的高度增长是对数级的。

rnk[x] 记录以x为根的树的高度上界（秩）。初始每个节点秩为0。

复杂度分析：反阿克曼函数

两个优化叠加之后的摊还复杂度，是这个数据结构最令人惊讶的地方。

1975年，Tarjan在论文 “Efficiency of a Good But Not Linear Set Union Algorithm” 中证明了：路径压缩与按秩合并结合，n次操作的总时间复杂度是O(n · α(n))，其中α是阿克曼函数的反函数（Inverse Ackermann Function）。

阿克曼函数A(m, n)是增长极端快的函数——A(4, 4)已经是一个远超宇宙中原子数量的天文数字。它的反函数α(n)增长极端慢：对实际使用中所有可能遇到的n（包括宇宙中原子的数量级），α(n) ≤ 4。

实践中，α(n)等同于常数。

所以并查集的每次操作摊还O(α(n)) ≈ O(1)，解决动态连通性问题是目前已知的最优算法（在某些计算模型下已被证明是下界）。

参考文献：

  • R. E. Tarjan, “Efficiency of a Good But Not Linear Set Union Algorithm,” Journal of the ACM, 22(2):215–225, 1975.
  • R. E. Tarjan and J. van Leeuwen, “Worst-case Analysis of Set Union Algorithms,” Journal of the ACM, 31(2):245–281, 1984.

经典变种

基础并查集只维护连通性，实际问题里经常需要更多信息。两个常见变种：

带权并查集

每个节点维护一个到父节点的”关系值” weight[x]，find 操作在路径压缩的同时更新权值（通常是某种可加或可乘的关系）。

经典例题：食物链（POJ 1182）。三类动物A、B、C，A吃B，B吃C，C吃A。给出一系列”x和y是同类”或”x吃y”的声明，判断哪些是假话。

用三倍扩点或者带权并查集的模3关系都能做，核心是在并查集里维护每个节点到根的”关系偏移量”。

可撤销并查集

普通路径压缩在回滚操作时很麻烦，因为它破坏了树的结构。如果需要支持撤销合并（如离线LCT问题、按时间分治），要把路径压缩换成只按秩合并，用一个栈记录所有修改，撤销时弹栈恢复。

代价是单次操作退化到O(log n)，但支持了撤销。这个权衡在 Offline Dynamic Connectivity 的分治框架里很常见。

做题 Checklist

拿到题目，判断是否能用并查集：

  • 涉及集合的合并和查询？-> 直接上
  • 图的连通性，边是静态或只有加边？-> 并查集
  • 需要维护到根的路径上某个值？-> 带权并查集
  • 边有撤销操作？-> 可撤销并查集或其他结构
  • 需要删边？-> 并查集无法直接支持，考虑离线倒序处理（把删边变成加边）或 Link-Cut Tree

板子题推荐：

  • POJ 1236, POJ 1182, POJ 2524
  • Codeforces: 731C, 566D（带权/带额外信息的变种）
  • LeetCode 547, 684, 685（入门练手）

说实话，每次在ACM比赛里遇到连通性问题，打出并查集板子的那一刻都会有一种奇怪的满足感。

三行 find，一个路径压缩赋值，摊还O(α(n))。

当年Tarjan证这个复杂度的时候，α函数这个东西大概在数学界也不算常见。但他硬是把它用上了，就为了精确描述这个数据结构有多快。

这种精确，是我喜欢算法的原因之一。芙宁娜大概也能理解，毕竟她是搞法律的，讲究的也是精确。
