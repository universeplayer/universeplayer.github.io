线段树：暴力美学的极致

发表于2026-03-04|更新于2026-03-04|算法
|总字数:2.2k|阅读时长:8分钟|浏览量:

为什么不能直接暴力

假设御坂美琴在学园都市的实验室里有一个长度为 $n$ 的数组，记录着每个实验时间段的电磁辐射读数。她需要频繁地回答两类问题：

  1. 区间 $[l, r]$ 内的辐射读数之和是多少？
  2. 把某段区间 $[l, r]$ 的读数全部加上一个偏移量 $\delta$（比如某次实验污染了一整段数据）。

暴力解法：每次 query 扫一遍区间，$O(n)$；每次 update 扫一遍区间，$O(n)$。如果有 $Q$ 次操作，总复杂度 $O(nQ)$。

$n = 10^5$，$Q = 10^5$，那就是 $10^{10}$ 次操作。

美琴的超电磁炮可以 $3 \times 10^8$ m/s，但服务器的 CPU 大约每秒跑 $10^8$ 次简单操作——$10^{10}$ 次，TLE 妥妥的。

线段树把两种操作都降到 $O(\log n)$，总复杂度 $O(Q \log n)$。对于上面的规模，从 100 秒变成 0.1 秒。这就是线段树存在的理由。

线段树的结构

线段树是一棵完全二叉树，每个节点维护一段区间 $[l, r]$ 上的信息。

  • 根节点覆盖整个数组 $[1, n]$
  • 每个内部节点 $[l, r]$ 的左子节点覆盖 $[l, \text{mid}]$，右子节点覆盖 $[\text{mid}+1, r]$，其中 $\text{mid} = \lfloor(l+r)/2\rfloor$
  • 叶子节点覆盖单个元素 $[i, i]$

对于一棵覆盖 $n$ 个元素的线段树，节点总数为 $O(n)$（准确说是最多 $4n$ 个，因此开 tree[MAXN * 4] 是惯例）。树高为 $O(\log n)$。

节点编号规则：根节点为 1，节点 $i$ 的左子节点为 $2i$，右子节点为 $2i+1$。这是用数组模拟树的标准做法，不需要指针，常数更小。

建树时间复杂度 $O(n)$——每个叶子访问一次，每个内部节点做一次合并操作。

区间查询

查询 $[l, r]$ 的区间和，递归策略如下：

当前节点覆盖区间 $[s, e]$：

  1. 如果 $[l, r]$ 完全包含 $[s, e]$，直接返回 tree[node]
  2. 如果 $[l, r]$ 与 $[s, e]$ 无交，返回 0
  3. 否则递归查询左右子节点，合并结果

时间复杂度分析： 每一层最多访问 4 个节点（直觉：查询区间的左右端点各自最多劈开 $O(\log n)$ 层，每层最多两个节点被”劈开”，其余要么完全包含要么完全不交）。严格证明可参考 Cormen et al., CLRS 4th ed., §14.1。结论：$O(\log n)$。

懒标记（Lazy Propagation）

这是线段树里最重要、也最容易写挂的部分。如果你在某道题上线段树WA了一半，大概率是懒标记写炸了。

问题： 区间更新怎么办？如果我要把 $[l, r]$ 上每个元素都加 $\delta$，暴力地对每个叶子做单点更新是 $O(n \log n)$，不可接受。

核心思路： 延迟更新。当我们确定某个节点 $[s, e]$ 完全被更新区间 $[l, r]$ 覆盖时，不递归下去更新子节点，而是把”这个节点的所有子节点都要加 $\delta$”这件事记录在该节点的 lazy 标记里。下次有操作需要访问子节点时，再把这个标记”下推”（push down）给子节点。

用芙宁娜的比喻来说明：芙宁娜作为枫丹法院的审判官，处理一批”枫丹全体居民税率上调 $\delta$”的政令。她不需要挨个通知每个居民，只需要在”枫丹”这个行政单元上打一个标记”待通知：+$\delta$”，等到有人真的来问具体某个地区的税率时，再把标记从上层往下推。

push_down 实现：

几个细节，逐一说清楚：

  1. tree[2*node] += lazy[node] * (mid - start + 1)：左子节点覆盖 $[\text{start}, \text{mid}]$，共 $\text{mid} - \text{start} + 1$ 个元素，每个加 $\delta$，所以子节点的区间和增加 $\delta \times \text{len}$。
  2. lazy[2*node] += lazy[node]：懒标记是累加的，不是覆盖。如果子节点已经有一个”待下推的+3”，现在又收到”+5”，它的新标记是”+8”，不是”+5”。
  3. push_down 之后把当前节点的 lazy 清零——该下推的已经下推了，不能让它再被重复计算。

区间更新：

懒标记的常见坑：

  • query 里忘了 push_down：你查询子节点之前必须先下推，否则子节点的 tree 值是过时的，查出来是错误答案。这是新手写线段树最经典的锅。
  • 懒标记叠加而非覆盖：lazy[child] += lazy[node]，不是 =。两种不同类型的更新（加法、乘法）混合时懒标记的合并更复杂，要仔细推导。
  • 叶子节点不需要 push_down：叶子没有子节点，push_down 在叶子上调用是无意义的（不会出错，但是白跑）。

扩展：更多玩法

权值线段树（离散化）： 线段树不只能维护”值的区间”，也可以维护”值域上的频率”。把所有值离散化后建线段树，可以支持第 $k$ 大、区间内值在 $[a, b]$ 范围的元素个数等查询。

线段树合并： 两棵线段树可以在 $O(n \log n)$ 时间内合并（对应节点的信息合并，只合并双方都存在的节点）。常用于树上问题：每个节点维护一棵线段树，自底向上合并。

主席树（可持久化线段树）： 在每次更新时只新建从根到叶子的 $O(\log n)$ 个节点，保留所有历史版本。可以回答”区间第 $k$ 大”这类带历史版本的查询。主席树展开讲够写一篇新文章，这里先埋个坑。

区间最大/最小值（RMQ）： 把 tree[node] = tree[left] + tree[right] 换成 max 或 min，维护区间极值。懒标记的更新语义也要相应修改（对 max 的懒标记通常是”区间赋值”而非”区间加法”，合并逻辑不同）。

做题 Checklist

碰到区间操作题：

  1. 操作是单点更新 + 区间查询？→ 基础线段树，不需要懒标记
  2. 操作是区间更新 + 区间查询？→ 懒标记必须有，push_down 不能漏
  3. 查询的信息能分治合并吗（和、最大值、最小值、GCD……）？→ 能，就能用线段树维护
  4. 空间开了 MAXN * 4 吗？→ 没开就 MLE 或越界
  5. 懒标记初始化为 0 了吗？→ 没初始化 bug 极难找
  6. query 里 push_down 有没有？→ 没有就 WA，而且是那种样例过了但 hack 一发就炸的 WA
  7. 合并操作写对了吗？→ 回溯时的 tree[node] = tree[left] + tree[right] 别漏

第 6 条不是玩笑。写了懒标记但 query 里忘了 push_down，样例里区间操作少的话完全测不出来，上了大数据就爆炸。美琴的超电磁炮可以击穿装甲，但击不穿一个忘了 push_down 的 WA。
